# Personal Website Workflow

本仓库是陈若非个人网站的静态站点，用于展示简历、产品案例、交互/UI 能力和 vibe coding/AI 产品化探索。后续迭代优先服务一个目标：让访问者在较短时间内理解“我是谁、做成过什么、如何思考和落地复杂产品”。

## Website Goals

网站的核心目标是建立一个可信、清晰、有作品证据的个人产品负责人作品集。

- 第一目标：让招聘方、合作方或同行快速理解陈若非的职业定位、核心能力和代表成果。
- 第二目标：用产品案例证明复杂业务系统、商业化增长、用户体验和 AI 工具化能力，而不是只展示履历摘要。
- 第三目标：沉淀个人方法论，让项目经验从“做过什么”升级为“如何判断、如何推进、如何拿到结果”。
- 第四目标：形成可持续迭代的个人网站工作台，后续能不断加入案例、设计证据、AI 原型和方法文章。

访问者进入网站后，应能在 30 秒内获得三个判断：

- 这是一个长期做 B 端电商 SaaS 和交易/商品/营销系统的人。
- 这个人既懂产品增长和商业化，也能处理复杂流程和体验设计。
- 这个人正在把 AI 和代码作为产品验证、表达和效率提升工具。

## Tech Stack

当前技术栈保持轻量静态站路线：

- HTML: 原生 HTML 页面，当前不使用框架和模板引擎。
- CSS: 单一 `styles.css` 管理全站变量、布局、组件、主题和响应式。
- JavaScript: 单一 `script.js` 管理轻量交互，包括导航、主题、移动端菜单、滚动状态和局部动效。
- Assets: 图片、SVG 和静态视觉资产放在 `assets/`。
- Source Materials: 原始案例文档、流程图、drawio 文件和研究素材放在 `素材/`。
- QA Notes: 视觉与功能验收记录写入 `design-qa.md`。

默认不引入构建工具、包管理器或前端框架。只有当某个明确目标需要更复杂能力时，才考虑引入新的技术依赖，并且要先说明收益、成本和迁移影响。

## Design Style

整体风格关键词：克制、专业、清晰、有作品感、可扫描。

- 视觉气质：接近成熟产品负责人作品集，而不是营销落地页或炫技型个人主页。
- 信息密度：保持中高信息密度，适合快速扫读；不要用大面积空泛 hero 稀释内容。
- 色彩：以当前蓝色强调体系为基础，允许每个产品案例有少量差异化 accent，但避免整站变成单一色系或高饱和装饰风。
- 排版：中文正文优先保证可读性，标题要有层级但不过度夸张；长段落要拆成有节奏的段落、列表或卡片。
- 组件：优先使用现有导航、卡片、内容区、项目卡片和 sticky 导航模式，新增组件要和现有圆角、阴影、边框、间距保持一致。
- 动效：只做辅助理解和反馈的轻量动效，不做影响阅读的复杂动画。
- 响应式：移动端不是桌面缩小版，要保证阅读顺序自然、触控区域足够、导航可用。
- 深色模式：所有新增页面和组件都必须兼容当前 `data-theme="dark"` 变量体系。

设计上避免：

- 空洞口号、过度抽象的渐变背景、装饰性堆叠卡片。
- 只有视觉冲击但没有项目证据的内容。
- 文本压图、文字重叠、横向滚动、按钮文字溢出。
- 为单个页面写一套完全孤立的视觉语言。

## Current Structure

- `index.html`: 简历首页，突出个人定位、核心经历、教育、技能和方法。
- `product-cases.html`: 产品案例入口页，目前以项目列表和视觉卡片为主。
- `interaction-ui.html`: 交互/UI 能力页，目前是能力摘要。
- `vibe-coding.html`: AI 与 vibe coding 探索页，目前是能力摘要。
- `styles.css`: 全站视觉、布局、主题和响应式样式。
- `script.js`: 导航、主题切换、移动端菜单、滚动高亮、案例页动效。
- `assets/`: 页面图片、SVG 与静态视觉资产。
- `素材/`: 原始内容资料、流程图、drawio 文件和项目案例素材。
- `design-qa.md`: 最近一次设计验收记录。

## Site Strategy

网站的信息架构保持四个一级入口：

- 简历：快速建立可信度，强调 B 端电商 SaaS、交易/商品/营销、商业化和 AI 产品化能力。
- 产品案例：展示深度项目成果，优先补齐掌中宝交易 OMS，再扩展商品、促销、工具吧、多卖宝。
- 交互/UI：展示复杂流程降噪、设计系统、转化页面和营销素材体系。
- vibe coding：展示用 AI 和代码做原型、验证产品判断、提升个人和团队效率的能力。

页面不做营销式空泛包装，优先清晰、克制、可扫描。视觉表达要帮助招聘方、合作方或产品负责人快速判断经验质量。

## Iteration Workflow

每次迭代按以下顺序推进：

1. 明确本轮目标
   - 判断本轮属于内容补齐、视觉改版、交互优化、性能/兼容修复，还是发布前 QA。
   - 每轮尽量只解决一个主要页面或一个明确链路。

2. 盘点素材
   - 优先阅读 `素材/` 中已有文档和图。
   - 对产品案例，先提炼项目定位、用户问题、关键决策、指标结果、个人贡献和可视化证据。
   - 对设计/UI 内容，先确认是否有截图、流程图、组件、前后对比或可重绘示意。

3. 内容结构先行
   - 先写页面大纲，再写正文。
   - 案例页建议结构：
     - 项目一句话定位
     - 背景与用户问题
     - 我的角色与责任
     - 关键产品决策
     - 核心流程或系统结构
     - 结果指标
     - 可迁移的方法论
   - 避免只罗列功能，要让读者看到判断、取舍和结果。

4. 页面实现
   - 优先复用现有 `site-header`、`resume-header`、`resume-content`、`resume-section`、`job-card` 等结构。
   - 新增样式先检查 `styles.css` 里是否已有可复用变量和组件。
   - 保持静态站可直接打开运行，不引入构建工具，除非本轮目标明确需要。

5. 视觉与响应式 QA
   - 至少检查桌面和移动端两个视口。
   - 重点确认导航、卡片、长中文文本、图片比例、深色模式、移动端侧滑菜单。
   - 页面不能出现横向滚动、文字重叠、按钮文字溢出或图片拉伸。
   - 验收结论更新到 `design-qa.md`。

6. 发布前整理
   - 保留源素材，但不要把临时文件误认为正式资产。
   - 新图片放入 `assets/`，原始研究/流程资料放入 `素材/`。
   - 提交前检查 `git status`，不要误删或覆盖未确认的用户素材。

## Code Requirements

HTML 要求：

- 页面使用语义化结构，优先使用 `header`、`main`、`section`、`article`、`nav`。
- 每个页面要有准确的 `title` 和 `meta description`。
- 导航、按钮、图片链接需要合理的 `aria-label` 或 `alt`。
- 新增详情页命名使用清晰英文，例如 `case-trade-oms.html`。
- 页面文案默认中文，必要时保留 SaaS、OMS、AI、Prompt、Agent 等英文术语。

CSS 要求：

- 优先复用 `:root` 变量和已有组件类，新增变量要有明确复用价值。
- 新样式要考虑浅色和深色主题，不硬编码只在单一主题可读的颜色。
- 固定格式元素要有稳定尺寸、比例或响应式约束，避免 hover、图片、长文本造成布局跳动。
- 避免大范围重构 `styles.css`，除非本轮目标就是整理样式系统。
- 修改后检查是否引入横向滚动、重叠、遮挡或移动端溢出。

JavaScript 要求：

- 保持原生 JavaScript，不引入依赖。
- 新交互必须渐进增强：脚本失败时核心内容仍可阅读。
- 新增逻辑要和现有菜单、主题、滚动监听共存，避免重复绑定全局事件。
- 需要滚动或观察视口时优先使用轻量、可读的实现，并注意性能。

资产要求：

- 页面正式引用的图片放入 `assets/`。
- 原始资料、流程图、drawio 和研究素材放入 `素材/`。
- 图片命名要能看出用途和页面，例如 `case-trade-oms-flow.png`。
- 不删除或清理未跟踪素材，除非用户明确要求。

Git 与协作要求：

- 修改前先看当前文件和 `git status`。
- 不覆盖用户已有改动，不清理无关未跟踪文件。
- 每轮改动范围要和目标一致，不顺手做无关重构。
- 重要页面改动后要更新 `design-qa.md`。

## Content Roadmap

优先级从高到低：

1. 掌中宝交易 OMS 深度案例
   - 基于 `素材/掌中宝交易OMS_项目成果_从产品定位到商业闭环.md`。
   - 配合订单系统流程、单据流转、钱货信息流转、泳道图等素材。
   - 目标是把“0-1、增长、商业化、复杂系统设计”讲透。

2. 产品案例列表升级为可进入详情的作品集
   - 每个项目卡片从锚点改为详情入口。
   - 先做掌中宝交易详情页，再决定其他案例是否做详情页或摘要页。

3. 交互/UI 页补充证据
   - 增加复杂流程前后对比、组件规范、转化页面案例。
   - 少讲“会设计”，多展示“如何让复杂系统更好用”。

4. vibe coding 页补充真实产出
   - 展示 AI 辅助产品分析、原型、评测、自动化工作流。
   - 突出产品负责人如何把 AI 放进业务闭环。

5. 首页简历精炼
   - 根据案例页完成度回调首页文案。
   - 首页只保留最强信号，把深入内容交给专题页。

## Acceptance Standards

每次交付从五个维度验收：目标、内容、视觉、交互、工程。

目标验收：

- 本轮改动服务明确目标，例如补齐一个案例、优化一个页面、修复一个体验问题。
- 页面重点足够清楚，读者不需要猜这个页面想证明什么能力。

内容验收：

- 内容逻辑完整，读者能看出背景、动作、结果和个人贡献。
- 案例内容必须包含可验证的结果或证据，例如指标、流程图、页面图、方法总结。
- 文案避免空泛形容词，优先使用具体场景、决策、取舍和结果。
- 中文表达自然，不出现明显机器翻译腔或中英文混乱。

视觉验收：

- 桌面和移动端布局稳定，无明显溢出、错位、遮挡。
- 浅色和深色模式均可读。
- 图片清晰、比例自然、加载后不破坏布局。
- 长标题、长中文段落、按钮文字和卡片内容不会溢出容器。

交互验收：

- 一级导航 active 状态正确。
- 移动端菜单可打开、关闭，遮罩和菜单项点击行为正常。
- 主题切换正常，并保持本地偏好。
- 锚点、返回顶部、滚动高亮等页面内交互符合预期。
- 键盘 Escape、点击外部关闭菜单等已有行为不被破坏。

工程验收：

- 页面引用的 CSS、JS、图片路径都能正常加载。
- 浏览器控制台无明显 JavaScript 报错。
- 没有引入不必要依赖或构建步骤。
- `git status` 中的变更可解释，不包含误删、临时文件或无关改动。
- `design-qa.md` 记录本轮检查结果。

## Definition Of Done

一次页面迭代完成需要同时满足：

- 本轮目标已经完成，没有留下影响主要体验的半成品。
- 已按 `Acceptance Standards` 完成目标、内容、视觉、交互和工程检查。
- 已更新必要文档或 QA 记录，尤其是重要页面改动后的 `design-qa.md`。
- 已确认 `git status` 中的变更范围合理，没有误碰用户素材或无关文件。

## Editing Notes

- 默认使用中文文案，保留必要英文术语如 SaaS、OMS、AI、Prompt、Agent。
- 文件内容优先保持 ASCII 标点，中文正文可自然使用中文字符。
- 不清理未跟踪素材，尤其是 `素材/` 下的 drawio、png、bkp 文件，除非用户明确要求。
- 修改 CSS 时优先使用现有变量，避免把网站改成单一色系或过度装饰。
- 如果新增详情页，命名建议使用 `case-trade-oms.html` 这类清晰英文文件名。
